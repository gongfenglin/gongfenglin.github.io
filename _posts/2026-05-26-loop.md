### From one-shot output to operating loop

The word **Loop** is useful because it pushes the discussion away from one-shot answers. A capable agent is not just a model call. It is a repeated operating cycle.

The loop I care about is simple: define the task, collect context, plan, act through tools, verify the result, record what happened, and feed the result back into the next run. Each cycle should make the next cycle less blind.

### The important part is verification

Many agent loops stop at "act." That is where reliability breaks. The loop becomes meaningful only when verification is part of the normal path, not an afterthought.

For coding agents, verification can be tests, builds, diffs, lint checks, benchmark runs, or human review gates. For LLM evaluation, it can be assertion suites, provider-level expected failures, XPASS reports, and trace inspection. For document pipelines, it can be conversion quality, latency, and retry behavior.

### Why I keep returning to this idea

Loop is a bridge between agent runtime design and evaluation design. It connects memory, tools, subagents, benchmarks, and release gates into one working rhythm.

My current work is mostly about making this loop explicit enough that an agent can be improved systematically instead of judged by scattered anecdotes.
